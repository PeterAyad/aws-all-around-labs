# Service Access Matrix

| Compute Service | To Access RDS (Postgres/MySQL)  | To Access S3 / DynamoDB (AWS APIs) | Reason (Why & The Golden Rule)                                                                                                                                                                                                     | How to? (Top-Level Steps for IAM)                                                                                                                                                                                                                          |
| :-------------- | :------------------------------ | :--------------------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **EC2**         | VPC Network + Connection String | **IAM Instance Profile**           | **Why:** Native AWS VM.<br><br>**Golden Rule:** RDS uses standard DB drivers (requires Network + Password). S3/DynamoDB use the AWS SDK (requires IAM identity).                                                                   | 1. Create an IAM Role with needed policies (e.g., S3 read).<br>2. Wrap it in an IAM Instance Profile.<br>3. Attach the profile to the EC2 instance via the console or CLI.                                                                                 |
| **Lambda**      | VPC Network + Connection String | **IAM Execution Role**             | **Why:** Native AWS Serverless.<br><br>**Golden Rule:** RDS needs a VPC attachment + DB credentials. AWS SDKs need AWS IAM credentials injected at runtime.                                                                        | 1. Create an IAM Role with needed policies.<br>2. Go to your Lambda function's "Configuration" tab.<br>3. Under "Permissions," assign this role as the "Execution Role."                                                                                   |
| **ECS**         | VPC Network + Connection String | **IAM Task Role**                  | **Why:** Native AWS Container Orchestrator.<br><br>**Golden Rule:** The container uses standard drivers for RDS, but the AWS SDK inside the container needs IAM permissions to sign API requests.                                  | 1. Create an IAM Role with needed policies.<br>2. Open your ECS Task Definition.<br>3. Assign this role as the **Task Role**.                                                                                                                              |
| **EKS**         | VPC Network + Connection String | **IRSA via OIDC**                  | **Why:** Non-AWS Orchestrator. EKS must "prove" to AWS who its internal pods are.<br><br>**Golden Rule:** Database drivers still just need network line-of-sight. The AWS SDK needs OIDC to bridge Kubernetes identity to AWS IAM. | 1. Create an OIDC provider for your EKS cluster in AWS IAM.<br>2. Create an IAM Role that trusts that OIDC provider.<br>3. Create a Kubernetes `ServiceAccount` and annotate it with the IAM Role's ARN.<br>4. Tell your Pod to use that `ServiceAccount`. |
